# PART 8: Java 17 & 21 现代特性 ⭐⭐ 2026 加分项

> 搜索发现:2026 年"一个 senior Java 开发者如果不能讨论 Java 17 特性,说明没跟上"。这些是加分项,尤其对 senior 岗位。

## 8.1 Record (Java 16+) ⭐

> 中文:Record 是不可变数据载体,自动生成构造器、getter、equals、hashCode、toString。一行代替一个 POJO。

```java
// 传统 POJO 要写几十行,Record 一行搞定
public record Transaction(String id, double amount, String currency) {}
// 自动有:构造器、id()、amount()、currency()、equals、hashCode、toString
Transaction tx = new Transaction("T1", 5000, "USD");
System.out.println(tx.amount());  // 5000
```

## 8.2 Sealed Classes (Java 17) ⭐

> 中文:密封类限制哪些类可以继承它,增强类型安全。

```java
public sealed interface Alert permits SanctionsAlert, FraudAlert {}
public final class SanctionsAlert implements Alert {}
public final class FraudAlert implements Alert {}
// 其他类不能实现 Alert
```

## 8.3 Pattern Matching for instanceof (Java 16+)

```java
// 旧写法
if (obj instanceof String) {
    String s = (String) obj;   // 还要强转
    System.out.println(s.length());
}
// 新写法 - 直接绑定变量
if (obj instanceof String s) {
    System.out.println(s.length());  // s 直接可用
}
```

## 8.4 Switch 表达式 (Java 14+)

```java
// 新的 switch 表达式,返回值,无需 break
String status = switch (alertType) {
    case "SANCTIONS", "PEP" -> "HIGH";
    case "FRAUD" -> "MEDIUM";
    default -> "LOW";
};
```

## 8.5 Virtual Threads / Project Loom (Java 21) ⭐⭐ 最新热点

> 搜索发现:这是 2026 最热的 Java 面试话题。面试官不指望你在生产用过,但要理解模型、局限、和 reactive 的关系。

**中文核心:**
> 传统的**平台线程(Platform Thread)**是 1:1 映射到操作系统线程的,每个约占 1MB 栈内存,创建几千个就会耗尽。这就是 thread-per-request 模型的瓶颈——线程大量时间在空等 I/O(数据库、外部服务),CPU 利用率低但线程却用光了。
>
> **虚拟线程(Virtual Thread)**是 JVM 管理的轻量级线程,不直接绑定 OS 线程。当虚拟线程遇到阻塞操作(I/O、sleep)时,JVM 把它从**载体线程(carrier thread)**上卸下,让载体线程去跑别的虚拟线程——OS 线程永远不空等。这样你能创建**几百万**个虚拟线程。

**English answer:**
"Virtual threads are lightweight threads managed by the JVM rather than the OS. Traditional platform threads map 1:1 to OS threads and cost about 1MB each, so you exhaust them at a few thousand. When a virtual thread hits a blocking operation like I/O, the JVM unmounts it from its carrier thread — a real OS thread — and mounts another virtual thread, so the OS thread never blocks. This lets you write simple synchronous blocking code that scales like async, without the complexity of reactive frameworks."

```java
// Java 21 - 每个任务一个虚拟线程,可以开百万个
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 1_000_000).forEach(i ->
        executor.submit(() -> {
            String data = restClient.getData(i);  // 阻塞 I/O
            dbRepository.save(data);               // 又一个阻塞 I/O
        })
    );
}
```

> **虚拟线程的局限(面试加分):**
> - CPU 密集型任务没帮助(它解决的是 I/O 等待,不是计算)
> - "pinning"问题:在 synchronized 块里阻塞会把虚拟线程钉在载体线程上
> - 依赖 native 阻塞的库(JNI)可能不兼容

> **和 CompletableFuture/reactive 的关系:** 虚拟线程让你用简单的同步阻塞代码达到接近 reactive 的扩展性,对 I/O 密集的 95% 应用来说,比复杂的 reactive 框架更划算。
